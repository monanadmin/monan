1#!/usr/bin/env python
2"""
3Reads the restart pointer named in the namelist file
4provided through the -f flag. If the file named by
5config_restart_timestamp_name does not exists, exits 0
6(meaning the run is not done). If the file does exist,
7checks that the current date does not exceed the end
8date specified by the -e flag with format
9YYYY-MM-DD_hh:mm:ss. (Note: only years between 0001
10and 9999 are supported.) If this date is exceeded,
11exits 1, indicating that the run should stop.
12Otherwise, exits 0, indicating the run has not yet
13completed.
14
15Note that this script will correctly detect that
16a simulation has completed only if a restart file is
17written at the desired end date. Otherwise, the
18script will detect that the simulation should be
19finished only after the first restart file is written
20out that exceeds the end date.
21
22An example snippet from a bash job script using this
23script to perform 12 month-long runs in sequence
24might look like the following:
25
26...
27runsPerJob=12
28endDate=0002-01-01_00:00:00
29
30for iter in (`seq 1 $runsPerJob`)
31do
32 ./check_progress.py -f namelist.ocean -e $endDate
33 if ($? != 0) then
34 echo 'run finished'
35 exit 1
36 endif
37
38 ./run.py
39
40 if ($? != 0) then
41 echo 'run failed'
42 exit 1
43 endif
44
45 ./setup_restart.py -f namelist.ocean -s "'file'"
46done
47
48In this example, the ./run.py script could perform a number
49of steps following the model run such as modifying forcing,
50updating the model state (e.g. for "offline" coupling),
51or performing sanity checks.
52
53A number of jobs could be chained with this script. If one of
54these jobs exits with exit status 1, this could be made to
55cancel dependent jobs (depending on the capabilities of the
56job scheduler). Even if not, the next job would immediately
57exit once it runs, thus using negligible computing time.
